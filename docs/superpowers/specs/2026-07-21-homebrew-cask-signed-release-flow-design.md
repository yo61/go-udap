# Homebrew cask release: GoReleaser generates, the tap lands it signed

## Context

`go-udap` ships to Homebrew as a **cask** in `yo61/homebrew-tap`
(`Casks/go-udap.rb`). Until now that cask was updated by GoReleaser
pushing the file **directly to the tap's `main`** via the REST Contents
API, at the end of `go-udap`'s tag-triggered `goreleaser.yaml` run.

That mechanism is now broken. On 2026-07-14 the tap's `Default Branch`
ruleset (`id 16783297`) was tightened and its App bypass actor removed —
a deliberate change recorded in the tap's
`decisions/2026-07-13-signed-commits-via-api-drop-bypass.md`. The ruleset
enforces four rules on `main`: `deletion`, `required_signatures`,
`pull_request` (0 required approvals), and `non_fast_forward`. A direct
push of an unsigned commit now violates both `pull_request` and
`required_signatures`:

```
homebrew cask: could not update "Casks/go-udap.rb":
PUT .../repos/yo61/homebrew-tap/contents/Casks/go-udap.rb: 409 Repository rule violations found
  Changes must be made through a pull request.
  Commits must have verified signatures.
```

Every `go-udap` release since — v2.4.4 (2026-07-20) and v2.4.5
(2026-07-21) — fails at the cask-push step. The binaries, SBOMs, and the
GitHub release still publish; only the tap update dies, so `brew` users
are pinned at 2.4.3.

The tap's **formulae** (`jobhound`, `unifictl`) were migrated to a
signed, PR-based flow when the bypass was dropped: a tap-side `bump-*`
workflow, triggered by `repository_dispatch` from each tool's release,
opens a `bump/*` PR whose commits are authored via GitHub's GraphQL
`createCommitOnBranch` (the only API path GitHub signs server-side — the
REST Contents API returns `verified=false`, confirmed empirically in the
tap's decision doc). `go-udap` is the one artifact left on the old
direct-push path. This is an **unfinished migration**, not a regression
to undo.

## Goal

Finish the migration for the cask: land `Casks/go-udap.rb` on the
protected `main` through a signed PR, with **no ruleset bypass** and **no
weakening of the tap's rules**, while keeping releases hands-off (as the
formulae are).

## Design overview

Split responsibilities along the line of what each repo authoritatively
knows:

- **`go-udap` owns cask *generation*.** GoReleaser already renders the
  full cask — version, the four platform `url`/`sha256` values, and the
  `manpage`/`completion`/`postflight` stanzas derived from the actual
  build. It keeps doing so, but **stops pushing**. The rendered cask is
  attached to the GitHub release as an asset. After the release
  publishes, `go-udap` fires a `repository_dispatch` at the tap.
- **The tap owns cask *landing*.** A new `bump-go-udap` workflow fetches
  the rendered cask from the release, commits it to `Casks/go-udap.rb`
  via `createCommitOnBranch` (signed), and opens a `cask/go-udap-<v>` PR.
  `tests.yaml` lints it; on green, a new `publish-cask` workflow
  squash-merges the PR through the merge API and deletes the branch.

This is a clean split, not the old direct push wearing a new coat:
GoReleaser **never touches the tap** — it only emits an artifact. The
tap is the sole writer of `main`, and every commit that lands there is
GitHub-signed.

### Why generation stays in `go-udap` (the shift-left decision)

The cask's `manpage`/`completion`/`postflight` stanzas are **derived
data** — a function of `go-udap`'s subcommand set. The rejected
alternative (tap owns a hand-maintained cask, edited in place per
release) would copy that derived list into the tap and let it **drift**:
if `go-udap` removed a subcommand, the tap's cask would keep a `manpage`
line pointing at a file no longer in the tarball, and `brew install`
would fail. Catching that downstream would mean adding an online
`brew audit --cask` special-case to the merge gate.

Generating the cask from the build — where the subcommand set is defined
— makes drift **structurally impossible** instead of monitored. There is
nothing to keep in sync, so the merge gate stays a plain static lint.
This is the "shift left" that motivated the design: fix the class of bug
at its source rather than guard against its symptom later.

## `go-udap` changes

### `.goreleaser.yaml`

- `homebrew_casks[0]`: set `skip_upload: "true"`. GoReleaser still
  renders `dist/homebrew/Casks/go-udap.rb` but does not commit or push
  it. Remove the now-unused `repository:` and `token:` blocks.
- `release.extra_files`: attach the rendered cask so the tap can fetch
  it:
  ```yaml
  release:
    draft: false
    use_existing_draft: true
    extra_files:
      - glob: ./dist/homebrew/Casks/go-udap.rb
  ```
  The asset lands at
  `https://github.com/yo61/go-udap/releases/download/v<version>/go-udap.rb`.
- The `# generated by GoReleaser. DO NOT EDIT.` header stays accurate —
  GoReleaser is still the generator; only the transport changed.

### `.github/workflows/goreleaser.yaml`

- **Remove** the `Style-fix the generated cask in the tap` step
  (clones the tap and `git push`es directly — it would fail the same
  ruleset). Its purpose — GoReleaser's cask emitting a trailing blank
  line `brew style` rejects — moves to the tap's lint gate; if the
  generated cask ever trips `brew style`, the fix is a GoReleaser cask
  template/formatting change, addressed in the plan's verification step.
- After the `goreleaser-action` step succeeds, add a step that fires the
  dispatch, reusing the existing App token (already scoped to
  `homebrew-tap` with `contents: write`, which the dispatch endpoint
  requires):
  ```yaml
  - name: Trigger tap cask bump
    env:
      GH_TOKEN: ${{ steps.tap_token.outputs.token }}
      VERSION: ${{ github.ref_name }}   # vX.Y.Z
    run: |
      set -euo pipefail
      gh api -X POST repos/yo61/homebrew-tap/dispatches \
        -f event_type=bump-go-udap \
        -f 'client_payload[version]='"${VERSION#v}"
  ```
- The `tap_token` can drop `go-udap` from its `repositories:` list — it
  is now used only to dispatch to `homebrew-tap`.

The dispatch's cross-repo contract is exactly `{event_type,
client_payload.version}`, matching the narrow contract the formula
bumps use.

## Tap changes

### Generalize `commit-formula-pr` → `commit-recipe-pr`

The existing composite action creates the `bump/*` branch, commits one
file via `createCommitOnBranch`, and opens the PR. Its only
recipe-specific detail is the hardcoded `Formula/<formula>.rb` path. Add
a `file-path` input and rename the action to `commit-recipe-pr` so it
serves both formulae and casks from one signing implementation (a single
source of truth for the signing dance is correctness-critical — the
decision doc treats getting the signing right as load-bearing).

- Update `bump.yaml`'s `uses:` and pass `file-path: Formula/<formula>.rb`.
- The cask caller passes `file-path: Casks/go-udap.rb`.

### `.github/workflows/bump-go-udap.yaml` (new)

Cask analog of `bump-unifictl.yaml`, but simpler — no PyPI, no
`update-python-resources`, no in-place editing:

- `on: repository_dispatch: types: [bump-go-udap]` and a
  `workflow_dispatch` with a `version` input (manual re-run path, as the
  formula bumps have).
- Resolve version: `client_payload.version || inputs.version`; regex-gate
  it before any shell use.
- Gate for no-op / already-open: skip if `Casks/go-udap.rb` already at
  this version, or if `cask/go-udap-<v>` already exists (mirrors
  `bump.yaml`'s gate).
- Fetch the rendered cask asset:
  `curl -fsSL https://github.com/yo61/go-udap/releases/download/v<v>/go-udap.rb -o Casks/go-udap.rb`.
  The asset already carries the version and the four `sha256`s GoReleaser
  computed, so there is no hash re-derivation and no `SHA256SUMS` parse.
- Mint the App token (contents + pull-requests write) and call
  `commit-recipe-pr` with `branch: cask/go-udap-<v>`,
  `file-path: Casks/go-udap.rb`.

Branch prefix is **`cask/`**, distinct from the formulae's `bump/`, so
`publish-bottles.yaml` (guarded on `startsWith(head_branch, 'bump/')`)
ignores cask PRs and the new `publish-cask.yaml` claims them. No edit to
`publish-bottles.yaml` is needed.

### `.github/workflows/publish-cask.yaml` (new)

`workflow_run` on `brew test-bot` completing, guarded like
`publish-bottles.yaml` but for the cask prefix:

```
github.event.workflow_run.conclusion == 'success' &&
github.event.workflow_run.event == 'pull_request' &&
github.event.workflow_run.head_repository.full_name == github.repository &&
startsWith(github.event.workflow_run.head_branch, 'cask/')
```

There is **no bottle step** — the signed commit already exists on the
branch, so this workflow only:

- validates the `cask/go-udap-<v>` branch name,
- finds the open PR for that head,
- squash-merges via `PUT /pulls/{n}/merge` with `merge_method=squash` and
  `sha=<branch head oid>` (the head-sha TOCTOU pin from the tap's
  `2026-07-10-pr-pull-head-sha-pin` decision — fail closed if the head
  moved), retrying a few times for async mergeability,
- best-effort deletes the branch (`DELETE /git/refs/heads/...`, ignoring
  404).

The squash commit is GitHub-signed, satisfying `required_signatures`;
the merge is a PR merge, satisfying `pull_request`. No bypass.

## Data flow

```
go-udap tag vX.Y.Z pushed
  └─ goreleaser.yaml
       ├─ GoReleaser: build, SBOMs, render cask (skip_upload) → dist/,
       │              publish release + attach go-udap.rb asset
       └─ dispatch → homebrew-tap: bump-go-udap {version}

homebrew-tap
  └─ bump-go-udap.yaml
       ├─ gate (version valid, not already current, branch not open)
       ├─ curl release asset go-udap.rb → Casks/go-udap.rb
       └─ commit-recipe-pr: createCommitOnBranch (signed) on
          cask/go-udap-<v>, open PR
  └─ tests.yaml (Casks/** path filter): --only-tap-syntax lint
  └─ publish-cask.yaml (on tests.yaml success, cask/ prefix)
       └─ squash-merge PR (signed), delete branch → main updated
```

## Alternatives considered

- **Add the release App back as a ruleset bypass actor.** Rejected: it is
  exactly what the tap's `2026-07-13-signed-commits-via-api-drop-bypass`
  decision deliberately removed. Reintroducing it reopens the
  "signed-main is a fiction maintained by an ever-present bypass"
  contradiction that work resolved.
- **Tap owns a hand-maintained cask, edited in place (version + 4
  sha256 via `awk`).** Rejected on the shift-left grounds above: it
  copies derived `manpage`/`completion` data into the tap and lets it
  drift, forcing a downstream online-audit special-case.
- **GoReleaser opens the PR itself (`homebrew_casks.pull_request`).**
  Rejected: GoReleaser's branch commit is unsigned; landing it would lean
  on squash-merge to launder the signature — the exact mechanism the tap
  decision doc examined and declined, preferring signed branch commits.
- **`go-udap` sends the four sha256s in the dispatch payload.** Rejected
  under a narrower question earlier: it widens the cross-repo contract
  and makes manual re-runs require pasting hashes. Superseded entirely by
  this design — the tap fetches the fully-rendered cask, so no hashes
  cross the boundary at all.

## Testing / verification

- **Tap workflows:** `actionlint` + `zizmor` clean (the tap gates on
  both). `commit-recipe-pr` generalization is exercised by the next real
  formula bump (regression) and the first cask bump (new path).
- **First real cask release:** the true end-to-end test, as the tap's
  reusable-bump decision notes for `workflow_call` resolution. Verify:
  release asset `go-udap.rb` present; `bump-go-udap` opens a
  `cask/go-udap-<v>` PR with a `verified` commit and no signature-block
  banner; `tests.yaml` green; `publish-cask` squash-merges; `main`'s
  `Casks/go-udap.rb` at the new version with a GitHub-signed commit;
  `brew upgrade go-udap` pulls it.
- **`brew style` on the generated cask:** confirm the GoReleaser-rendered
  cask passes `brew style`/`--only-tap-syntax` without the old
  post-push `--fix` step. If it does not, fix the GoReleaser cask
  formatting so the artifact is clean at generation time (shift-left,
  again) rather than reintroducing a downstream fixer.

## Rollout

The tap's bypass is already gone, so there is no safety net to preserve —
the current flow is already fully broken (v2.4.4/v2.4.5). Sequence:

1. Land the tap changes (`commit-recipe-pr`, `bump-go-udap.yaml`,
   `publish-cask.yaml`).
2. Land the `go-udap` changes (`.goreleaser.yaml`, `goreleaser.yaml`).
3. Cut the next `go-udap` release (a patch is sufficient) and verify the
   end-to-end flow above.
4. Backfill: the tap is stuck at 2.4.3 while the release is 2.4.5. Once
   the flow is proven, trigger `bump-go-udap` via `workflow_dispatch`
   with `2.4.5` to land the current version without waiting for a new
   release.

A decision doc will be added to the tap's `decisions/` journal
(`2026-07-21-cask-signed-release-flow.md`) recording this, per the tap's
convention and building on
`2026-07-13-signed-commits-via-api-drop-bypass`.

## Open risks

- **Dispatch endpoint permission.** The design assumes the App token's
  `contents: write` on `homebrew-tap` authorizes
  `POST /repos/.../dispatches`. Confirm during implementation; widen the
  token's scope minimally if the endpoint needs more.
- **Asset availability timing.** The dispatch fires after the
  `goreleaser-action` step, which publishes the release and uploads
  `extra_files` before returning — so the `go-udap.rb` asset is present
  when the tap fetches it. Verify ordering holds on the first real run.
- **`skip_upload` vs. required `repository`.** GoReleaser may still
  require `homebrew_casks[].repository` to be present for schema/render
  even when `skip_upload: "true"`. If so, keep a minimal `repository:`
  block (and drop only the `token:`), rather than removing it. Confirm
  with `goreleaser release --snapshot` during implementation.
